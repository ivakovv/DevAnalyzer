# Документация проекта: DevAnalyzer

## 1. Введение

### 1.1 Назначение документа
Данный документ описывает архитектуру, функциональные требования и сценарии использования веб-приложения **DevAnalyzer**. Приложение предназначено для автоматического анализа профилей разработчиков (GitHub) и предоставления рекрутерам инструментов для оценки кандидатов.

### 1.2 Название проекта
**DevAnalyzer** (Приложение для анализа профиля кандидата)

### 1.3 Краткое описание
Система позволяет пользователям авторизоваться, запрашивать анализ GitHub-аккаунта, просматривать отчеты. HR-специалисты (рекрутеры) имеют расширенный функционал: сохранение истории просмотров, избранное, сравнение кандидатов, добавление комментариев и экспорт данных. Администратор управляет пользователями системы.

## 2. Функциональные требования

### 2.1 Роли пользователей

| Роль | Описание | Доступный функционал |
|------|----------|---------------------|
| **Неавторизованный пользователь (Guest)** | Пользователь, не прошедший аутентификацию | • Доступ к главной странице<br>• Форма авторизации/регистрации<br>• Анализ GitHub недоступен |
| **Пользователь (User)** | Обычный зарегистрированный пользователь | • Базовый анализ GitHub-аккаунта<br>• Просмотр своих отчетов<br>• Управление профилем |
| **HR / Рекрутер (HR)** | Специалист по подбору персонала | • Весь функционал User<br>• История просмотров отчетов (с пагинацией)<br>• Избранное (сохранение кандидатов)<br>• Подробная аналитика по кандидату<br>• Сравнение кандидатов<br>• Комментарии к кандидатам<br>• Экспорт отчета в PDF<br>• Интеграция с Telegram |
| **Администратор (Admin)** | Управляющий системой | • Полный доступ к системе<br>• Управление пользователями (CRUD)<br>• Просмотр системных логов и метрик |

### 2.2 Основные функции (Must Have)

#### 2.2.1 Авторизация и аутентификация
- Регистрация нового пользователя
- Вход в систему (логин/пароль)
- Восстановление пароля по email с использованием JWT токена
- Управление профилем (изменение данных, смена пароля, загрузка аватарки)

#### 2.2.2 Анализ профиля
- Прием ссылки на GitHub профиль кандидата
- Прием и парсинг резюме (PDF/DOCX) с использованием нейросети
- Генерация отчета о навыках (стек технологий)
- Отслеживание Open Source контрибуций

#### 2.2.3 Управление данными (HR)
- История просмотров отчетов с пагинацией
- Избранное (сохранение кандидатов в отдельный список)
- Сравнение кандидатов (сопоставление данных из отчетов)
- Добавление комментариев к кандидатам

### 2.3 Дополнительные функции (Should Have)
- **Интеграция с Telegram:** уведомления о статусе анализа, отправка краткого отчета
- **Экспорт в PDF:** генерация профессионального отчета о кандидате
- **Система оценивания:** расчет рейтинга кандидата на основе кода и активности

## 3. Архитектура системы

### 3.1 Общая архитектура
Проект использует **микросервисную архитектуру**. Коммуникация между сервисами осуществляется через REST API. В качестве точки входа используется **API Gateway**.

## 8. Безопасность

| Механизм | Описание | Реализация |
|----------|----------|------------|
| **Аутентификация** | JWT токены, хранящиеся в HttpOnly cookies | Токен передается через Gateway и валидируется перед каждым запросом к микросервисам. Срок жизни токена: 24 часа |
| **Хранение паролей** | BCrypt хеширование с солью | Используется BCryptPasswordEncoder с strength factor = 10. Пароли никогда не хранятся в открытом виде |
| **Аватарки** | Прямые ссылки на S3 с ограниченным TTL | GET ссылки — 1 час, PUT ссылки — 5 минут. Предотвращает несанкционированный доступ к файлам |
| **Ролевая модель** | Доступ на уровне Gateway на основе роли | Роль извлекается из JWT, проверяется перед проксированием запроса к микросервисам |
| **CORS** | Ограничение доступа к API | Разрешены только доверенные домены (frontend domain) |
| **Rate Limiting** | Ограничение количества запросов | Реализовано на уровне Gateway: 100 запросов/минуту для авторизованных пользователей, 20 запросов/минуту для неавторизованных |
